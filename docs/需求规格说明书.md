## 一、项目概述

### 1.1 项目背景

本人作为中国移动通信集团江苏有限公司数字化部集客业务产品经理，日常核心工作涵盖：集客业务销售能力建设、生产数据运营保障、需求管理、投诉处理、会议协调及知识沉淀等多个条线。虽然已有公司级标准化系统（需求管理系统、投诉支撑系统、OA系统等），但缺乏一个以个人工作流为核心、跨系统聚合信息、并能主动沉淀业务知识的统一管理平台。

目前已具备以下基础能力：
- **需求催办系统**（`D:\项目\需求催办系统`）— Node.js 后端 + MySQL，已运行，6000+条需求数据
- **统一邮件中心**（`D:\项目\统一邮件中心`）— HTTP API 邮件发送服务，端口 3210
- **Obsidian 知识库**（`D:\项目\知识图谱`）— 本地 Markdown 知识管理工具，600+需求卡片，21类家客产品知识
- **Chrome 浏览器插件** — 需求信息提取 + 催办邮件发送

> **架构整合说明（2026-07-16）**：上述分散系统已于 2026-07-16 全部并入 PMWB 统一入口，不再作为独立系统运行——需求催办系统的网页功能（催办中心/邮件记录/集团需求）并进 PMWB 前端；Chrome 插件改连 PMWB 后端（`/api/v1/plugins/*`），原 2525 本地中继退役；3000 端口服务可停。详见「模块8：统一邮件催办」。

### 1.2 建设目标

1. **一个入口，全面管理**：通过本地浏览器访问的 Web 系统，统一管理需求、运营、会议、待办、知识等全部工作内容。
2. **数据驱动，主动可视**：关键指标、任务进度、风险预警在首页看板一目了然，从“被动响应”转向“主动管控”。
3. **流程闭环，知识沉淀**：需求从提出到交付、运营从发现问题到解决问题、会议从计划到决议执行，全流程留痕，并自动沉淀为结构化知识资产。
4. **易于扩展，面向 AI**：积累的业务数据与知识可供未来 AI 助手分析利用。

### 1.3 系统名称与标识

| 维度 | 命名 | 说明 |
| :--- | :--- | :--- |
| 系统中文名 | **产品经理个人工作台** | 正式名称 |
| 系统英文标识 | **`pmwb`** | 数据库表前缀、代码命名空间 |
| 邮件模板类型 | **`pmwb_{业务类型}`** | 如 `pmwb_reminder`、`pmwb_meeting` |

### 1.4 目标用户

| 用户角色 | 使用场景 | 核心诉求 |
| :--- | :--- | :--- |
| 产品经理（本人） | 日常工作的统一入口 | 掌控全局、减少遗漏、快速沉淀 |
| 外部协作者（可选） | 共享进度、查看报告 | 只读或有限协作，非本阶段重点 |

### 1.5 使用场景

- **每日上班**：打开首页看板，查看今日待办、超期事项、今日会议。
- **需求跟踪**：查看需求列表，点击需求查看关联开发工单、邮件催办记录、交付物归档。
- **运营问题处理**：录入或跟踪生产运营问题，记录根因与解决方案，沉淀为 Bug 解决方案或运维 SOP。
- **会议后**：新建会议纪要，生成待办并关联到责任人和相关需求/工单。
- **周/月复盘**：通过统计报表查看需求处理量、工单上线情况、运营问题闭环率。

## 二、系统架构

### 2.1 总体架构

```
+------------------+      +-------------------+      +-----------------+
|   Chrome 插件     |      |   Python 定时脚本  |      |   Obsidian Vault |
| (需求采集+         | ---> | (采集/清洗/聚合)    |      | (Markdown知识库) |
|  开发工单采集)      |      |                   |      +--------+--------+
+------------------+      +-------------------+               |
        |                         |                           |
        v                         v                           v
+---------------------------------------------------------------+
|              产品经理个人工作台 Python Web 后端                   |
|           (FastAPI — RESTful API + Markdown读写)               |
+----------+----------------------------+------------------------+
           |                            |
           v                            v
+------------------+      +-----------------------------+
|   MySQL yxtyg_db  |      |   统一邮件中心 (localhost:3210)|
| (sent_emails 已有 )|      |   POST /api/send             |
| (pmwb_* 新增)     |      |   POST /api/templates/render |
+------------------+      +-----------------------------+
```

### 2.2 技术选型

| 层次 | 技术 | 说明 |
| :--- | :--- | :--- |
| 后端框架 | FastAPI | 现代异步 Python Web 框架，自动生成 OpenAPI 文档 |
| 前端框架 | Vue 3 + Element Plus | 组件丰富，适合管理后台类系统 |
| 数据库 | MySQL（yxtyg_db） | 复用现有库，新增 `pmwb_*` 表 |
| 文件/知识库 | Obsidian Vault | 本地 Markdown 存储，通过后端读写 |
| 邮件服务 | 统一邮件中心 | HTTP API 调用，不直接连 SMTP |
| 定时任务 | APScheduler / 系统任务计划 | 用于数据采集、提醒、聚合 |
| 部署方式 | 本地运行 | 个人使用，localhost 访问，可后续扩展 |

### 2.3 邮件服务

所有邮件发送统一通过统一邮件中心 HTTP API，不使用直接 SMTP 连接。

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/send` | POST | 发送邮件 |
| `/api/templates/:id/render` | POST | 渲染邮件模板 |

模板命名规范：`pmwb_{业务类型}`，如：
- `pmwb_reminder` — 需求催办
- `pmwb_meeting_invite` — 会议邀请
- `pmwb_meeting_minutes` — 会议纪要分发
- `pmwb_deliverable_notice` — 交付物通知

## 三、模块规划

### 模块0：首页看板

**定位**：系统首页，聚合所有关键信息，提供“今日工作”总览。

**Obsidian 联动**：保留 `00-工作台/工作台看板.md` 作为离线备用版。

**数据来源**：MySQL 各表聚合（待办数/超期数/需求各状态统计/工单进度/会议日程）。

**功能清单**：

| 功能 | 说明 |
| :--- | :--- |
| 今日待办 | 展示今日到期或优先级高的待办，支持快捷完成 |
| 超期预警 | 红色高亮显示已逾期的待办、工单、运营问题 |
| 今日会议 | 展示今日会议列表，快速进入纪要 |
| 关键指标卡 | 需求总数、在途工单数、未闭环运营问题数、本周会议数 |
| 需求状态分布 | 饼图/柱状图展示需求各状态占比 |
| 工单进度跟踪 | 最近活跃的开发工单列表及进度 |
| 快捷入口 | 新建待办、新建工单、新建会议纪要、发送催办 |
| 通知中心 | 系统提醒、邮件发送结果、定时任务状态 |

### 模块1：待办中心

**定位**：管理个人所有待办任务，支持与需求、工单、会议、运营问题关联。

**功能清单**：

| 功能 | 说明 |
| :--- | :--- |
| 待办创建 | 标题、内容、分类、优先级、截止日期、重复周期、提醒时间 |
| 待办分类 | 工作分类：需求跟进、工单跟踪、运营问题、会议准备、学习、其他 |
| 优先级 | P0 / P1 / P2 / P3，P0 默认置顶并红色标记 |
| 状态管理 | 待处理 / 进行中 / 已完成 / 已取消 |
| 业务关联 | 可关联到需求、开发工单、运营问题、会议 |
| 重复任务 | 支持按日/周/月重复，如每周运营周报 |
| 提醒通知 | 到期前提醒，首页看板超期高亮 |
| 视图切换 | 列表视图、看板视图（按状态）、日历视图 |
| 筛选排序 | 按分类、优先级、状态、截止日期筛选排序 |

**数据表**：`pmwb_todo`

```sql
CREATE TABLE pmwb_todo (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '自增ID',
    title VARCHAR(255) NOT NULL COMMENT '待办标题',
    content TEXT COMMENT '待办内容',
    category VARCHAR(64) DEFAULT 'other' COMMENT '分类：requirement/ticket/operation/meeting/study/other',
    priority ENUM('P0','P1','P2','P3') DEFAULT 'P2' COMMENT '优先级',
    status ENUM('todo','in_progress','done','cancelled') DEFAULT 'todo' COMMENT '状态',
    due_date DATE COMMENT '截止日期',
    due_time TIME COMMENT '截止时间（可选）',
    remind_at DATETIME COMMENT '提醒时间',
    repeat_type ENUM('none','daily','weekly','monthly') DEFAULT 'none' COMMENT '重复类型',
    related_type VARCHAR(64) COMMENT '关联对象类型：requirement/ticket/operation/meeting',
    related_id VARCHAR(64) COMMENT '关联对象ID',
    source VARCHAR(64) COMMENT '来源：manual/meeting/plugin',
    completed_at DATETIME COMMENT '完成时间',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status (status),
    INDEX idx_due_date (due_date),
    INDEX idx_related (related_type, related_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='待办任务表';
```

### 模块2：需求管理

**定位**：在现有 `sent_emails` 表基础上，做需求信息的集中展示、增强、跟踪与统计。

**现有数据**：`sent_emails` 表 6000+条需求记录。

**功能清单**：

| 功能 | 说明 |
| :--- | :--- |
| 需求列表 | 展示所有需求，支持分页、搜索、筛选、排序 |
| 需求详情 | 点击需求查看完整信息、催办历史、关联工单、交付物、会议纪要 |
| 状态跟踪 | 需求状态：已提出 / 已受理 / 开发中 / 已上线 / 已关闭 / 已暂停 |
| 催办记录 | 展示该需求历次催办邮件发送记录（来自 `email_records`） |
| 关联工单 | 展示该需求关联的所有开发工单 |
| 标签/备注 | 为需求添加个人标签、内部备注 |
| 快速催办 | 在需求详情页一键发送催办邮件 |
| 需求统计 | 按状态、按系统、按月份统计需求数量 |
| 版本要求 | 为需求设置「版本要求」日期（需求管理要求的上线时间），页面手选；作为关联开发工单进度跟踪的基准 |

**数据表**：复用 `sent_emails` 作为主数据源，新增扩展表 `pmwb_requirement_ext` 存储个人管理字段。

```sql
CREATE TABLE pmwb_requirement_ext (
    id INT AUTO_INCREMENT PRIMARY KEY,
    req_id VARCHAR(64) NOT NULL COMMENT '需求编号，对应 sent_emails.req_id',

    owner_note TEXT COMMENT '负责人备忘',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_req_id (req_id),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='需求管理扩展表';
```

### 模块4：业务运营监控

**定位**：跟踪生产运营问题、数据异常、一线投诉，形成“发现 → 处理 → 复盘 → 沉淀”闭环。

**功能清单**：

| 功能 | 说明 |
| :--- | :--- |
| 问题录入 | 问题标题、类型、发现时间、影响范围、关联系统、关联需求/工单 |
| 问题类型 | 数据异常 / 系统故障 / 一线投诉 / 流程阻塞 / 性能问题 / 其他 |
| 状态跟踪 | 待处理 / 处理中 / 待验证 / 已解决 / 已关闭 / 已挂起 |
| 处理记录 | 记录每次排查过程、根因分析、解决方案 |
| 影响评估 | 影响客户数、影响业务量、是否需上报 |
| 关联沉淀 | 解决问题后，可一键生成 Bug 解决方案或运维 SOP 知识条目 |
| 统计报表 | 按问题类型、按系统、按月份统计问题数量与闭环率 |
| 重点关注 | 超期未解决、影响重大的问题在首页高亮 |

> **分类说明（2026-07-16 重构）**：运营监控采用「1 张主表 + 5 大类」结构。`pmwb_operation_issue` 新增 `category` 字段区分 5 大类——`bug`(BUG管理) / `data`(数据异常管理) / `prod`(生产问题分析) / `task`(临时交办任务) / `complaint`(热点投诉)；原 6 种 `issue_type`（data_error/system_failure/complaint/process_block/performance/other）降级为「子类」，每类页内仍可选。每类页顶部含「闭环率」= (已解决+已关闭)/总数，并突出责任人 `handler`。

**数据表**：`pmwb_operation_issue`

```sql
CREATE TABLE pmwb_operation_issue (
    id INT AUTO_INCREMENT PRIMARY KEY,

    source VARCHAR(64) COMMENT '来源：monitor/complaint/oa/manual',

    impact_scope TEXT COMMENT '影响范围',

    root_cause TEXT COMMENT '根因分析',
    solution TEXT COMMENT '解决方案',
    related_req_id VARCHAR(64) COMMENT '关联需求编号',
    related_ticket_no VARCHAR(64) COMMENT '关联开发工单编号',
    related_system VARCHAR(128) COMMENT '关联系统',
    obsidian_path VARCHAR(512) COMMENT '沉淀知识条目路径',
    is_overdue TINYINT(1) DEFAULT 0 COMMENT '是否超期',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_status (status),
    INDEX idx_type (issue_type),
```

### 模块6：知识库

**定位**：将系统内沉淀的需求、工单、运营问题、会议等资产，与 Obsidian 知识库打通，形成可检索、可复用的知识条目。

**功能清单**：

| 功能 | 说明 |
| :--- | :--- |
| 知识条目列表 | 展示知识库中所有条目，按分类、标签筛选 |
| 知识分类 | 产品档案 / 运营知识 / 需求资产 / 会议资产 / 个人知识 / 学习知识 |
| 标签体系 | 统一标签管理，支持多标签 |
| 全文检索 | 搜索知识条目标题与内容 |
| 来源关联 | 知识条目可关联到需求、工单、运营问题、会议 |
| 模板创建 | 按模板新建产品档案、运维 SOP、Bug 解决方案等 |
| 快速查看 | 在工作台内直接预览 Markdown 内容 |
| 目录同步 | 按 Obsidian 目录结构读写，保持两边一致 |

**数据表**：`pmwb_knowledge_item`

```sql
CREATE TABLE pmwb_knowledge_item (
    id INT AUTO_INCREMENT PRIMARY KEY,
    item_id VARCHAR(64) NOT NULL COMMENT '知识条目编号',
    title VARCHAR(255) NOT NULL COMMENT '标题',
    category VARCHAR(64) NOT NULL COMMENT '分类：product/operation/requirement/meeting/personal/study',
    sub_category VARCHAR(128) COMMENT '子分类',
    tags VARCHAR(512) COMMENT '标签，逗号分隔',
    obsidian_path VARCHAR(512) NOT NULL COMMENT 'Obsidian 文件路径',
    source_type VARCHAR(64) COMMENT '来源类型：requirement/ticket/operation/meeting/manual',
    source_id VARCHAR(64) COMMENT '来源对象ID',
    summary TEXT COMMENT '摘要',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_item_id (item_id),
    INDEX idx_category (category),
    INDEX idx_source (source_type, source_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='知识库条目表';
```

### 模块7：产品圣经（ProductBible）

**定位**：内置产品档案/运维知识检索与编辑入口，实时读取 Obsidian 知识库作为单一事实源，不落库。

**功能清单**：

| 功能 | 说明 |
| :--- | :--- |
| 业务切换 | 顶部业务切换器，按 `PRODUCT_BIBLE` 配置加载业务（当前已接入「集团短信业务」，新增业务仅加一行配置） |
| 文档阅读 | 左侧 TOC + 正文渲染；\`\`\`mermaid\`\`\` 自动成图；宽表横向滚动；**H2 章节可折叠/展开其下 H3 子章节** |
| 关键字查询高亮 | 左侧搜索框输入关键词，正文命中处黄色高亮 + 计数(1/N) + ↑↓ 翻页 + × 清除 |
| 在线编辑写回 | 「编辑内容」进入 Markdown 源码框，保存经确认后写回 Obsidian vault 对应 `.md`（后端 `PUT /api/v1/product-bible/{key}`） |

**数据来源**：Obsidian Vault（`D:\项目\知识图谱`），业务域→路径映射在 `backend/core/config.py` 的 `PRODUCT_BIBLE` 列表。无新增数据库表。

### 4.2 三阶段迁移策略

1. **上线前**：保持现有结构完全不变

## 六、非功能性需求

### 6.1 性能需求

- 首页看板加载时间 ≤ 2 秒
- 列表页查询 ≤ 1 秒（单次查询 ≤ 1000 条）
- 邮件发送异步处理，不阻塞前端操作

### 6.2 安全需求

- 本地部署，不对外暴露服务
- 数据库连接使用本地配置，不硬编码密码
- 文件上传限制类型和大小，避免安全风险

### 6.3 可用性需求

- 界面响应式，支持 1920×1080 及以上分辨率
- 操作有明确反馈：成功提示、失败提示、加载状态
- 支持键盘快捷键，提升操作效率

### 6.4 可维护性需求

- 代码模块化，业务逻辑与数据访问分离
- 统一错误处理和日志记录
- 数据库变更通过迁移脚本管理

### 6.5 可扩展性需求

- 预留 AI 分析接口，未来可接入大模型做智能分析
- 模块之间低耦合，新增模块不影响已有模块
- 邮件模板、知识模板可配置，不硬编码
